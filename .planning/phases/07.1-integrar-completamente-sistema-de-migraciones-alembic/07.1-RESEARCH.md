## Summary

Alembic 1.12.1 is already installed in the Python environment (as a transitive dependency) but is not configured for this project. The project currently uses `Base.metadata.create_all()` in `models/database.py::init_db()` to create all 32 tables, plus a standalone one-off migration script at `migrations/add_invite_link_to_channels.py`. The standalone migration has already been executed (the `invite_link` column exists in the `channels` table). Phase 07.1 must: (1) initialize the Alembic configuration pointing to the existing `models.database.Base` and current `DATABASE_URL`, (2) generate an initial migration via `alembic revision --autogenerate` to capture the current schema as the baseline, (3) mark the standalone migration as already applied (or incorporate its column), (4) add alembic to `requirements.txt`, and (5) update Railway's deployment to run migrations.

**Primary recommendation:** Use `alembic init alembic` with `target_metadata = models.database.Base.metadata`, configure `alembic.ini` to read `DATABASE_URL` from the environment, and generate the baseline migration with `--autogenerate`. The standalone `add_invite_link_to_channels.py` migration is a historical artifact and should be archived (not deleted) after Alembic takes over.

### Pattern 1: Alembic + SQLAlchemy 2.0 Base
**What:** Standard Alembic initialization that points to the existing `models.database.Base` metadata.
**When to use:** Always, for this project.
**Example:**
```python
# alembic/env.py
from logging.config import fileConfig
from sqlalchemy import engine_from_config, pool
from alembic import context

import sys
from pathlib import Path
sys.path.insert(0, str(Path(__file__).parent.parent))

from dotenv import load_dotenv
load_dotenv()

from models.database import Base
from config.settings import bot_config

config = context.config
config.set_main_option("sqlalchemy.url", bot_config.DATABASE_URL)

if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata
```
**Source:** [Alembic docs — Migration Environment](https://alembic.sqlalchemy.org/en/latest/tutorial.html)

### Pattern 2: Baseline Migration via Autogenerate
**What:** Generate the initial Alembic revision from the current live schema (not from zero).
**When to use:** When Alembic is introduced to an existing database.
**Why critical:** The database already has 32 tables created by `Base.metadata.create_all()`. The baseline migration must capture this existing state so Alembic knows the current version.

```bash
# 1. Initialize alembic
alembic init alembic

# 2. Edit alembic/env.py to point to models.database.Base (see Pattern 1)

# 3. Generate baseline revision
alembic revision --autogenerate -m "initial schema baseline"
```

**Important:** The autogenerate will detect differences between `models/models.py` and the live DB. Since the migration `add_invite_link_to_channels.py` has already run, the model and DB should be in sync — the autogenerated revision should show 32 tables and no "pending" column changes. The revision should be marked as the baseline.

**Pitfall:** If autogenerate detects the `invite_link` column as a "new" column (because the model has it but the baseline revision doesn't), subsequent `alembic upgrade head` would try to add it again. **Solution:** Either (a) mark the baseline migration as already applied (`alembic stamp head`), or (b) ensure the initial revision includes all current tables.

### Pattern 3: Standalone Migration Integration
**What:** Handle the existing `migrations/add_invite_link_to_channels.py`.
**Decision:** Archive the file (move to `migrations/archive/`) rather than delete. The column has already been added to the DB, so there is nothing to "migrate" — it was a one-off script. After Alembic is set up, this file is no longer needed.

### Pattern 4: Railway Deployment with Alembic
**What:** Run Alembic migrations on Railway after each deployment.
**When to use:** Production deployments.

```bash
# In Railway's deployment command or start script:
alembic upgrade head
```

**Alternative (safer):** Run migrations before starting the bot process:
```bash
alembic upgrade head && python bot.py
```

Railway's PostgreSQL environment variable `DATABASE_URL` must be set. Alembic's `env.py` reads it via `bot_config.DATABASE_URL`, which loads from the `.env` file or Railway's environment.

### Anti-Patterns to Avoid
- **Using `create_all()` after Alembic is set up:** This would bypass Alembic's version tracking. Remove or disable `Base.metadata.create_all(bind=engine)` calls in `init_db()` once migrations are in place.
- **Using autogenerate on a dirty migration:** Always run `alembic upgrade head` before `alembic revision --autogenerate` to avoid conflicting branch heads.
- **Manual SQL in migrations without `op.execute()` wrapper:** Raw SQL strings should use `op.execute(text("SQL"))` in Alembic.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Schema versioning | Custom migration tracking table | Alembic `alembic_version` table | Standard, reliable, community-accepted |
| Adding columns | Standalone Python scripts | `alembic revision -m "add column" && alembic upgrade head` | Idempotent, version-tracked, rollbackable |
| Baseline schema setup | Manually writing all 32 tables in SQL | `alembic revision --autogenerate` | Captures exact current schema from models |
| Deployment migrations | Railway raw SQL runs | `alembic upgrade head` in startup | Standard, consistent, version-safe |

**Key insight:** Alembic's `alembic_version` table tracks which revision is applied. `Base.metadata.create_all()` does not interact with this table, so they can coexist — but the project should eventually disable `create_all()` and rely entirely on Alembic for schema changes.

## Common Pitfalls

### Pitfall 1: Baseline migration detects all tables as "new additions"
**What goes wrong:** `alembic revision --autogenerate` sees the DB has 32 tables and the revision is empty, so it generates `CREATE TABLE` for all of them. When deployed to a fresh DB, it works. But if someone later runs `alembic upgrade head` on a production DB that already has all tables, Alembic tries to create them again (and fails on conflict).
**Why it happens:** The autogenerated revision includes `op.create_table()` for all 32 tables. Running `alembic upgrade head` on an already-migrated DB causes "table already exists" errors.
**How to avoid:** After creating the baseline revision, stamp it as applied on existing databases: `alembic stamp 001_initial --sql` or run `alembic stamp <revision>` on each existing DB (SQLite dev + Railway PostgreSQL). New deployments get `alembic upgrade head`.
**Warning signs:** "sqlite3.IntegrityError: UNIQUE constraint failed" or "psycopg2.errors.DuplicateTable" on deployment.

### Pitfall 2: SQLAlchemy 2.0 enum types not rendered correctly in autogenerate
**What goes wrong:** Enum columns (e.g., `Enum(ChannelType)`, `Enum(TokenStatus)`) may render differently between SQLite and PostgreSQL in autogenerate diffs.
**Why it happens:** SQLite stores enums as VARCHAR, PostgreSQL uses native ENUM types. Autogenerate may try to modify columns that are semantically the same.
**How to avoid:** Alembic's `render_item()` hook can be configured in `env.py` to render enums as VARCHAR for consistency, or use `include_object` to skip enum-only changes.
**Warning signs:** `alembic revision --autogenerate` shows enum-related changes with no actual schema difference.

### Pitfall 3: Mako template not installed
**What goes wrong:** `alembic init` fails or migration files are empty.
**Why it happens:** Mako is a dependency of alembic but if the environment has a broken install.
**How to avoid:** Verify `pip show mako` returns a version. Alembic 1.12.1 includes Mako as a dependency.

### Pitfall 4: `alembic.ini` hardcodes a database URL
**What goes wrong:** Works in dev (SQLite) but fails in Railway (PostgreSQL) because the URL is hardcoded.
**Why it happens:** Default `alembic.ini` template has `sqlalchemy.url = driver://user:pass@localhost/dbname`.
**How to avoid:** Set `sqlalchemy.url` from the environment in `env.py` using `config.set_main_option()` as shown in Pattern 1.

## Open Questions

1. **Should `init_db()` be removed or disabled after Alembic is set up?**
   - What we know: `init_db()` is called in `bot.py::on_startup()`. It runs `create_all()` on every startup.
   - What's unclear: Whether anything in the codebase calls `init_db()` for initial setup only, or relies on it always running.
   - Recommendation: Keep `init_db()` as a fallback (only runs `create_all()` if tables don't exist via SQLAlchemy's default behavior) but add a comment. Or better: replace it with `alembic upgrade head` in the startup sequence.

2. **Should the standalone migration file be deleted or archived?**
   - What we know: `migrations/add_invite_link_to_channels.py` was a one-off script, already executed.
   - What's unclear: Whether any documentation or documentation references this file.
   - Recommendation: Archive to `migrations/archive/add_invite_link_to_channels.py` and add a README pointing to `alembic/`.

3. **SQLAlchemy 2.0 deprecation warnings with Alembic autogenerate?**
   - What we know: Project uses SQLAlchemy 2.0.28 with legacy `declarative_base()` from `sqlalchemy.ext.declarative`.
   - What's unclear: Whether Alembic 1.12.1 fully supports SQLAlchemy 2.0's new ORM APIs in autogenerate.
   - Recommendation: Use `target_metadata = Base.metadata` which is the supported approach. Avoid using `from sqlalchemy.orm import DeclarativeBase` (new 2.0 style) since the project uses the classic style.
